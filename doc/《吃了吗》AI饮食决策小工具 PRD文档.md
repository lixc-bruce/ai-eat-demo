# 《吃了吗》AI饮食决策小工具 PRD文档（V2.0）

## 一、产品概述

### 1.1 产品名称

吃了吗 - AI每日饮食决策小工具

### 1.2 产品定位

一款轻量化AI饮食决策工具，专为每日纠结「吃什么、怎么吃、吃健康」的用户打造。摒弃复杂食谱、繁琐筛选，依托AI智能匹配场景、口味、身体状态，快速给出个性化饮食方案，解决当代人高频饮食选择焦虑，主打**极速决策、贴合日常、简单易用**。

### 1.3 目标用户

核心用户：每日为饮食选择发愁的人群，精准覆盖上班族、学生党、独居人群、选择困难症用户。

用户痛点：每日三餐反复纠结、不知道适配当下场景吃什么、想吃好吃的又怕不健康、懒得思考搭配、外卖/家常菜选择两难。

### 1.4 核心产品价值

1. 节省决策时间：AI快速终结饮食纠结，无需反复思考、翻看攻略；
2. 个性化适配：贴合口味、场景、饮食偏好、身体状态推荐，不推送千篇一律的食谱；
3. 轻量化无负担：无复杂操作、无冗余功能，主打快速解决刚需问题。

## 二、核心功能（3个核心功能 + 用户系统）

### 2.1 核心功能1：AI一键三餐智能推荐

#### 功能描述

用户选择用餐时段（早餐/午餐/晚餐/加餐），勾选自身状态（饿感、口味偏好、忌口、场景），AI实时生成专属饮食方案，包含家常菜、外卖可选清单。

#### 功能细则

1. 支持自定义筛选条件：
   - 口味：清淡/重口/酸甜/麻辣/不选
   - 忌口：香菜/葱姜/海鲜/乳糖不耐/坚果/不选（可多选）
   - 需求：减脂/增肌/养胃/解馋/不选
   - 场景：快速搞定/认真吃饭/外卖速食/居家做饭/不选
2. 每次推荐给出「3套可选方案」：
   - 方案A：极简速食方案（5-15分钟搞定）
   - 方案B：家常正餐方案（营养均衡搭配）
   - 方案C：解馋轻食方案（满足口腹之欲）
3. 每套方案输出格式固定：方案名称、推荐菜品列表（含简要做法/外卖搜索关键词）、预估耗时、热量区间。
4. 支持「重新推荐」按钮，不满意可重新生成。

#### 验收标准

| 编号 | 标准 | 验证方式 |
|------|------|----------|
| AC1-1 | 用户完成时段+偏好选择到看到结果，核心路径≤3次点击 | 手动测试 |
| AC1-2 | AI返回的每套方案必须包含：名称、菜品列表、预估耗时、热量区间，缺一不可 | 接口断言 |
| AC1-3 | 方案中的菜品为国内常见食材/外卖平台可获取品类，不推荐小众进口食材 | 抽检审核 |
| AC1-4 | 所有偏好条件留空时也能正常生成推荐（走默认大众口味兜底） | 自动化测试 |
| AC1-5 | 重新推荐生成的新方案与上一轮不重复（由prompt层面约束） | 手动测试 |
| AC1-6 | AI接口超时或返回异常格式时，前端展示统一降级方案（见5.5节） | 自动化测试 |

#### 边界与约束

- 单用户每日免费推荐次数：**20次**（含重新生成），超出后提示「今日推荐次数已用完，明天再来吧」
- 两次推荐间隔≥2秒时显示loading，<2秒不显示（防抖处理）
- 推荐结果不做持久化存储（仅收藏时写入），减少存储成本和隐私风险

### 2.2 核心功能2：饮食疑问AI实时问答

#### 功能描述

针对用户饮食相关疑问，提供轻量化AI答疑，解决用户选择方案后的延伸问题。

#### 功能细则

1. 支持高频疑问：食物相克、热量估算、食材替代、懒人做法、外卖避坑、经期/胃病/减脂饮食禁忌等；
2. 对话轻量化，支持短句、口语化提问；
3. 自动留存近期20条问答记录，方便用户二次查看（更早记录归档至历史表）；
4. 每次问答记录关联到当前会话（session_id），支持断点续聊。

#### 验收标准

| 编号 | 标准 | 验证方式 |
|------|------|----------|
| AC2-1 | 用户提问到收到回复耗时≤15秒（含网络延迟） | 压测 |
| AC2-2 | AI回复末尾必须附带免责声明：「以上内容仅供参考，不构成医疗建议」 | 接口断言 |
| AC2-3 | 问答记录按时间倒序展示，最近20条实时可查 | 手动测试 |
| AC2-4 | 敏感问题（自残、极端减肥、危险偏方）触发安全拦截，返回固定安全提示文案 | 自动化测试 |

#### 边界与约束

- 单次问答输入上限500字，超出前端截断提示
- 单用户每日问答次数上限：**30次**
- 问答记录保存策略：近20条热数据存Redis，全量存MySQL

### 2.3 核心功能3：今日饮食收藏回顾

#### 功能描述

用户可收藏满意的AI饮食推荐方案，自动归档留存，支持每日/每周回顾，形成个人专属饮食库。

#### 功能细则

1. 每套推荐方案下方自带收藏按钮（星形icon），一键保存；
2. 收藏页面按日期分组展示，每天一个分组，组内按时间倒序；
3. 支持单条删除，长按或左滑触发删除确认；
4. 无数量限制。

#### 验收标准

| 编号 | 标准 | 验证方式 |
|------|------|----------|
| AC3-1 | 点击收藏后0.5秒内按钮状态变更为已收藏（乐观更新） | 手动测试 |
| AC3-2 | 收藏列表按日期正确分组，跨天数据不混淆 | 自动化测试 |
| AC3-3 | 删除收藏有二次确认弹窗，确认后不可恢复 | 手动测试 |
| AC3-4 | 未登录状态点击收藏，引导跳转登录页 | 手动测试 |

#### 边界与约束

- 收藏数据持久化到MySQL，Redis缓存用户最近7天收藏（加速首页加载）
- 删除为软删除（is_deleted=1），保留数据用于后续可能的恢复需求

---

### 2.4 用户系统

#### 功能描述

基于手机号+短信验证码的轻量登录系统，支撑收藏、偏好设置、问答记录等需要用户标识的功能。

#### 功能细则

1. 输入手机号 → 获取短信验证码（60秒冷却）→ 输入验证码 → 登录/注册（无感，首次自动注册）
2. 登录态使用JWT Token，有效期7天，过期需重新登录
3. 支持退出登录

#### 验收标准

| 编号 | 标准 | 验证方式 |
|------|------|----------|
| AC4-1 | 短信验证码4分钟内有效，超时提示重新获取 | 自动化测试 |
| AC4-2 | 同一手机号60秒内不可重复发送验证码 | 自动化测试 |
| AC4-3 | JWT Token过期后API请求返回401，前端自动跳转登录页 | 自动化测试 |

## 三、整体用户流程

### 3.1 核心主流程

```
打开Web页面 → [未登录] 浏览首页（可体验推荐但不可收藏）
            → [已登录] 选择用餐时段 → 勾选偏好（可跳过）
            → 点击「AI生成今日吃法」
            → 展示3套饮食方案（骨架屏loading，流式输出）
            → 用户浏览方案 → 收藏 / 换一版 → 完成当日饮食决策
```

### 3.2 分支流程

| 场景 | 流程 |
|------|------|
| 推荐不满意 | 点击「重新推荐」→ 重新调用AI → 生成全新方案（前端替换当前结果） |
| 查看历史收藏 | 底部Tab「收藏」→ 按日期分组展示 → 可删除 |
| 饮食疑问 | 底部Tab「问答」→ 输入问题 → AI回复 → 自动记录 |
| AI接口超时(>15s) | 前端终止请求 → 展示降级兜底方案 → 提示「AI响应超时，已为您推荐今日热门搭配」 |
| AI接口报错(5xx) | 自动重试1次 → 仍失败则展示降级方案 + 提示「服务繁忙，请稍后再试」 |
| 网络断开 | 离线提示toast → 恢复后自动重连（仅在推荐页和问答页生效） |
| 短信发送失败 | 提示「验证码发送失败，请检查手机号或稍后再试」 |
| JWT过期 | API返回401 → 前端拦截器清除token → 跳转登录页 → 用户无感重新登录 |

### 3.3 降级兜底方案

当AI接口完全不可用时，前端展示预设的静态兜底方案（按用餐时段内置3套通用搭配），确保用户总能获得基本服务——这是区别于竞品的核心体验保障。

## 六、数据模型设计

### 6.1 数据库表设计（MySQL 8.0）

#### 用户表 `eat_user`

```sql
CREATE TABLE eat_user (
    id          BIGINT AUTO_INCREMENT PRIMARY KEY,
    phone       VARCHAR(11)  NOT NULL UNIQUE COMMENT '手机号（加密存储）',
    nickname    VARCHAR(50)  DEFAULT '' COMMENT '用户昵称',
    avatar_url  VARCHAR(255) DEFAULT '' COMMENT '头像URL',
    created_at  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_phone (phone)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户表';
```

#### 用户偏好表 `eat_user_preference`

```sql
CREATE TABLE eat_user_preference (
    id              BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id         BIGINT NOT NULL UNIQUE COMMENT '关联用户ID',
    default_taste   VARCHAR(50)  DEFAULT '' COMMENT '默认口味：light/heavy/sweet_sour/spicy',
    default_taboos  VARCHAR(255) DEFAULT '' COMMENT '默认忌口，逗号分隔',
```

## 八、AI调用策略与成本控制

### 8.1 模型选择与配置

| 项目 | 配置 |
|------|------|
| 首选模型 | 通义千问 Qwen-Plus（性价比优先） |
| 备选模型 | DeepSeek-V3（通义不可用时自动切换） |
| 调用方式 | HTTP POST，兼容OpenAI chat/completions接口格式 |
| 超时时间 | 连接超时5s，读取超时15s |
| 重试策略 | 1次自动重试（间隔1s），仅对5xx错误重试 |

### 8.2 Prompt策略

**饮食推荐System Prompt要点：**
- 角色：你是一位资深营养搭配师和中国家庭厨师
- 输出格式要求：严格按JSON Schema返回（方便后端解析）
- 约束：推荐菜品使用国内常见食材，标注外卖可搜索关键词
- 安全边界：不推荐极端饮食方式，不推荐生食/未煮熟食物

**问答System Prompt要点：**
- 角色：你是一位饮食健康顾问
- 输出末尾固定追加免责声明
- 敏感词拦截由后端前置处理（不依赖AI自身过滤）

### 8.3 成本估算（MVP阶段）

| 项目 | 估算 |
|------|------|
| 每次推荐token消耗 | prompt ~500 tokens + 输出 ~800 tokens = ~1300 tokens |
| 每次问答token消耗 | prompt ~300 tokens + 输出 ~400 tokens = ~700 tokens |
| 预估DAU | 100人 |
| 人均日推荐次数 | 5次 |
| 人均日问答次数 | 3次 |
| 日Token消耗 | 100 × (5 × 1300 + 3 × 700) = 860,000 tokens |
| 月Token消耗 | ~26M tokens |
| 月成本（Qwen-Plus，百万token约2元） | ~52元/月 |
| 月成本（DeepSeek-V3，百万token约1元） | ~26元/月 |

### 8.4 成本优化措施

1. **结果缓存**：相同入参（时段+口味+忌口+需求+场景组合）1小时内复用缓存，命中率预估40%
2. **次数限制**：单用户每日推荐20次、问答30次，防滥用
3. **模型降级**：高峰期可自动切换到更低成本的模型（如Qwen-Turbo）
4. **静态兜底**：AI完全不可用时返回预设的热门方案，保证基本可用

## 九、异常处理与降级策略

### 9.1 异常场景汇总

| 异常场景 | 前端表现 | 后端处理 |
|----------|----------|----------|
| AI接口超时(>15s) | 终止请求，展示兜底方案 + toast"AI响应超时，已为您推荐热门搭配" | 超时后返回fallback数据 |
| AI接口返回5xx | toast"服务繁忙，请稍后再试"，展示上次成功缓存结果 | 重试1次→仍失败→记录告警日志→返回兜底 |
| AI返回格式解析失败 | 展示兜底方案 + 后台记录异常日志 | catch解析异常后返回兜底，同时写错误日志 |
| Redis不可用 | 功能降级：推荐/问答仍可用（直接查MySQL+调AI） | 捕获Redis连接异常，跳过缓存逻辑 |
| MySQL不可用 | 全局提示"系统维护中"，所有写操作返回503 | 返回503，触发前端全局错误页 |
| 短信服务不可用 | 登录页提示"验证码服务暂不可用，请稍后再试" | 记录告警，返回503 |
| 用户触发限流 | toast"操作太频繁，请稍后再试" | 返回429状态码+Retry-After头 |
| JWT过期 | 前端拦截器静默清除token→跳转登录页 | 返回401 |
| 内容安全拦截 | 展示固定文案"请提出健康的饮食问题" | 检测敏感词→不调AI→直接返回安全提示 |

## 十、法律合规与数据安全

### 10.1 免责声明

1. **推荐结果免责**：所有饮食方案展示区域末尾固定展示「以上推荐仅供参考，不构成健康或医疗建议」
2. **问答免责**：AI问答每次回复末尾自动追加「以上内容仅供参考，不构成医疗建议。如有健康问题请咨询专业医生」
3. **用户协议**：首次启动弹窗展示《用户协议》和《隐私政策》，需用户勾选同意后方可使用

### 10.2 数据安全

| 数据项 | 安全措施 |
|--------|----------|
| 手机号 | MySQL存储使用AES-256加密，日志中脱敏为138****1234 |
| 短信验证码 | Redis存储，4分钟过期，验证后立即删除 |
| JWT Secret | 环境变量注入，不写入配置文件，长度≥256位 |
| 用户偏好/收藏 | 仅用户本人可访问（基于JWT中userId做数据隔离） |

| 日志规范 | 不记录用户手机号明文、不记录问答完整内容（只记ID和字数） |
